# 怡天科技官网(www.etia-tech.com)代码审计报告

- **审计日期**:2026-07-02
- **审计对象**:中国区官网现有代码库
  - `etia-web-api`(后端 API,Laravel Lumen)
  - `etia-boss-web-admin`(内容管理后台 CMS,Vue 2)
- **未覆盖范围**:官网**前台展示层代码**未提供,故前台的 SEO、首屏性能、移动端适配未纳入本次评估(见第 6 节)。
- **当前状态**:线上仍可正常访问运行。

---

## 0. 结论摘要(TL;DR)

**评估结论:需要重建,但不是紧急重建。**

现有系统"功能可用",但**整套技术栈已全部停止维护(EOL)、构建链已经无法正常运行、且生产密钥已泄露**。这不是"代码写得差"的问题,而是"这套资产已无法被安全地长期维护"的问题。

建议采取**两步走**:

1. **立即安全止血(1–2 天,与后续路线无关,必须做)**:轮换全部泄露密钥、关闭生产 Debug、移除代码库中的 `.env`、给登录/发信接口加限流。
2. **按计划重建(建议 6–12 个月内启动,非救火)**:在支持周期内的技术栈上重写,数据库结构可直接迁移,内容零丢失。

不建议"原地长期维修":整条技术栈没有一个组件还在维护窗口内,连一个能跑的构建环境都需要先抢救。

## 1. 系统架构概览

| 代码库 | 作用 | 技术栈 | 规模 |
|---|---|---|---|
| `etia-web-api` | 官网后端接口 | PHP + Laravel **Lumen 7** | 首页/标签行业/全系产品/设备行业/新闻/关于/下载/发信 等模块 |
| `etia-boss-web-admin` | 官网内容管理后台 | **Vue 2.5** + Element-UI + vue-cli 3 | 约 140 个 `.vue` + 170 个 `.js`,8 个业务模块,约 2.1 万行 |

架构模式:后台 CMS(SPA)→ 调用 API → 读写数据库 → 前台展示层渲染内容。属于典型的**内容驱动型企业官网**,功能边界清晰、复杂度中等。

---

## 2. 技术栈老化情况(重建的主要理由)

| 组件 | 当前版本 | 状态 | 影响 |
|---|---|---|---|
| Vue | 2.5.17 | ❌ **已 EOL(2023-12-31)** | 官方不再发布任何安全补丁 |
| Element-UI | 2.15.5 | ❌ 停止开发 | 仅支持 Vue2,已被 Element Plus 取代 |
| @vue/cli | 3.x | ❌ 已废弃 | 现行 5.x,整体已被 Vite 取代 |
| node-sass | 4.14.1 | ❌ 已废弃 | 依赖旧版 Node,**新版 Node 上无法安装编译** |
| axios | 0.18.0(2018) | ❌ 多个已知 CVE | SSRF、ReDoS、凭证泄露等 |
| Laravel Lumen | 7.x | ❌ **已 EOL** | Laravel 7 安全支持 2021 年已结束 |
| PHP | 7.2 / 7.3 | ❌ **已 EOL(2020 / 2021)** | 运行时无任何安全更新 |
| nginx(Docker 镜像) | 1.13-alpine(2017) | ❌ 大量 CVE | 基础镜像过旧 |

**判断:从前端框架、构建工具、HTTP 客户端,到后端框架、语言运行时、部署镜像,没有一个组件仍在维护窗口内。** 意味着即使不重建,也无法获得任何官方安全补丁。

## 3. 安全问题清单(按严重程度排序)

### 🔴 严重

**S1. 生产环境密钥已泄露**
`etia-web-api/src/.env.prod` 被打包进源码,含**明文真实生产密钥**:
- 数据库密码、Redis 密码
- **腾讯云 COS 的 SecretId / SecretKey**(可操作对象存储,危害最大)
- 邮箱发信密码、`APP_KEY`

> 根因:`.gitignore` 只忽略了 `.env`,未覆盖 `.env.prod` / `.env.dev` 变体,导致其被提交。
> **处置:无论是否重建,应立即轮换/重置以上全部凭证,并将所有 `.env*` 移出代码库。**

**S2. 生产环境开启 `APP_DEBUG=true`**
出错时向外部访客暴露堆栈、文件路径、配置等信息——信息泄露,且便于攻击者探测。

### 🟠 高

**S3. 密码存储使用 `md5(密码 + 盐)`**(`AdminService::encryptionPwd`)
MD5 不适合存储密码,可被快速暴力/彩虹表破解。应改用 `bcrypt` 或 `argon2`。

**S4. 存在拼接式原生 SQL(潜在注入面)**(`BaseModel::updateBatch`)
该方法将字段值直接字符串拼接进 `UPDATE ... CASE WHEN` 原生 SQL,无参数绑定/转义。
> 现状:目前仅被"分类排序"接口(`SortController`)调用,传入的是排序数字,**当前调用点实际风险较低**;但方法本身不安全,任何未来复用都会引入真实注入漏洞。应改为参数绑定实现。

### 🟡 中

**S5. 公开发信接口无防护**(`POST /app/email/save`)
- 无验证码、无频率限制;
- `email` / `mobile` 仅校验"非空",未校验格式;
- 易被批量刷接口、当作垃圾邮件跳板。

**S6. 后台登录无防暴力破解**
登录接口无失败锁定、无限流。经核查,**全项目未使用任何 throttle / 限流中间件**。

**S7. dev 环境存在免密调试后门**(`Authorization` 中间件)
当 `APP_ENV=dev` 且请求带 `api-debug: true` 头时可绕过登录。仅限 dev,但一旦生产环境被误配为 dev 即形成完全绕过,较脆弱。

## 4. 工程与可维护性问题

**E1. 构建链已失效(关键)**
`Dockerfile` 与依赖配置指向 **`registry.npm.taobao.org`(淘宝 npm 源)**,该源已下线(迁至 npmmirror.com);叠加 `node-sass 4` 在新版 Node 上无法编译。
> **实际后果:今天按原样重新构建/部署本项目,大概率直接失败。** 这也是"原地维修"性价比极低的核心原因——需先抢救出一个能跑的构建环境。

**E2. 后端几乎没有自动化测试**
`tests/` 仅含 Lumen 默认脚手架(`ExampleTest`、`TestCase`),无有效业务测试。任何改动都缺乏回归保障。

**E3. 依赖锁定与镜像绑定过时基础设施**
`nginx:1.13-alpine`、旧版 node 基础镜像、废弃的镜像源等,均不利于在现代 CI/CD 中复现构建。

## 5. 重建 vs 翻新 —— 决策分析

| 维度 | 原地翻新(升级现栈) | 重建 |
|---|---|---|
| 技术栈可维护性 | ❌ Vue2 / Lumen7 / PHP7 均 EO,升级即等于重写 | ✅ 落在支持周期内 |
| 构建链 | ❌ 需先抢救失效的构建环境 | ✅ 全新现代工具链 |
| 安全补丁 | ❌ 无官方补丁可用 | ✅ 持续可获得 |
| 数据迁移成本 | — | ✅ 低,表结构可直接沿用 |
| SEO / 首屏 | 取决于未知的前台 | ✅ 可用 SSR 一并优化 |
| 工作量 | 因跨大版本升级,实际接近重写 | 中等,功能边界清晰 |

**核心判断**:由于 Vue2→Vue3、Lumen7→Laravel11、PHP7→PHP8 都是**破坏性大版本跨越**,"升级"的实际工作量已经接近"重写";而重建还能顺带解决构建链、SEO、测试缺失等问题。**因此重建的综合性价比高于翻新。**

同时,这是一个功能中等、数据模型清晰(现有 API 路由已完整定义内容结构)的企业官网,**重建的迁移风险和工期都可控**。

## 6. 建议的重建技术选型(供后续讨论)

| 层 | 建议 | 说明 |
|---|---|---|
| 前台展示 | **Nuxt 3 / Next.js(SSR)** | 官网重 SEO 与首屏,SSR 优于纯 SPA |
| 后台 CMS | **Vue 3 + Vite + Element Plus** | 界面逻辑可大量复用现有后台 |
| 后端 API | **Laravel 11 + PHP 8.3** | 现有 Controller/Service/Model 分层清晰,可平移改写 |
| 数据库 | 沿用现有结构 | 直接迁移,内容零丢失 |
| 部署 | 现代化 CI/CD + 新版基础镜像 | 替换失效镜像源与过旧 nginx |

> 说明:官网前台代码本次未提供,若为纯 SPA,则对搜索引擎收录不利——建议在重建时改为 SSR。此项待拿到前台代码后确认。

## 7. 行动建议(分阶段)

**阶段 0 — 安全止血(立即,约 1–2 天,与是否重建无关)**
1. 轮换全部泄露密钥:数据库、Redis、腾讯云 COS、邮箱、`APP_KEY`(对应 S1)
2. 生产关闭 `APP_DEBUG`(S2)
3. 将所有 `.env*` 移出代码库,修正 `.gitignore`(S1)
4. 给登录与发信接口加限流 + 发信接口加验证码(S5、S6)

**阶段 1 — 决策与准备(1–2 周)**
5. 确认前台代码现状与 SEO 需求;确定重建范围与优先级
6. 数据库结构梳理与迁移方案

**阶段 2 — 重建实施(按计划推进,建议 6–12 个月内启动)**
7. 按第 6 节选型重写前台 / 后台 / API,数据迁移,灰度切换
